## 🔴 Hallazgo crítico (corregido en esta corrida)

### Columnas inexistentes en `PROFILE_PUBLIC_COLS` → habría roto swipe/matches/perfiles

En el commit `b9736bc` agregué a `PROFILE_PUBLIC_COLS` 6 columnas (`company_uniqueness`, `company_photos`, `company_positions`, `seniority_levels`, `company_tags`, `linkedin_url`) para cerrar un gap de visibilidad. **Pero esas columnas NO existían en la tabla `profiles`.**

Impacto si se hubiera regenerado el APK:
- `db.profiles.getPublicById`, `getDiscovery` y `getWithProfiles` usan `select(PROFILE_PUBLIC_COLS)`. Supabase **rechaza** un select que pide una columna inexistente con error `column profiles.X does not exist`.
- Resultado: **swipe roto, matches rotos, perfiles públicos rotos** — los flujos centrales de la app.

Causa de fondo más profunda: el **onboarding de empresa** (`useOnboardingCompany`) y mi nuevo **editor de empresa** ya guardaban esas columnas vía `db.profiles.create`. Como no existían, esos guardados fallaban silenciosamente. El código entero asumía columnas que el schema no tenía.

**Fix**: migración `016_company_profile_missing_columns` — `ALTER TABLE profiles ADD COLUMN` para las 6 (TEXT para uniqueness/linkedin, JSONB para las listas, consistente con culture_values/company_benefits/company_tech_stack que ya eran jsonb). Verificado: las 6 columnas ahora existen.

> Este es exactamente el tipo de bug que la **sección 6 (Schema integrity)** del qa-auditor existe para atrapar — y lo atrapó **antes** de que regeneraras el APK.
