## рҹ”ҙ Bugs crГӯticos

### #3 вҖ” WelcomeView dark mode: textos invisibles

**ReproducciГіn**: `localStorage.talently_dark_mode='true'` + recargar `/`.

**SГӯntoma**: heading "Talently" y "Encuentra tu" son casi invisibles. CTA "Inicia sesiГіn" tiene contraste pobre.

**Causa probable**: `WelcomeView.jsx` define color de heading hardcoded en CSS o el token `--text-primary` no se invierte correctamente en `[data-theme='dark']` para esta vista especГӯfica.

**Severidad**: рҹ”ҙ вҖ” primera impresiГіn de la app rota en dark mode.

### #4 вҖ” Mensajes duplicados en chat al enviar

**ReproducciГіn**: enviar cualquier mensaje vГЎlido en `/app/messages/<matchId>`.

**SГӯntoma**: el mensaje aparece dos veces:
1. Una vez como optimistic update (id `temp-<timestamp>`)
2. Una segunda vez cuando llega el INSERT echo via realtime (UUID real)

El check `if (prev.some((m) => m.id === newMessage.id))` en `useMessages.js:36-39` no detecta el duplicado porque los IDs son distintos.

**Fix**: en el handler de realtime, reemplazar el optimistic (Гәltimo mensaje con `_optimistic: true` del mismo sender y contenido idГ©ntico) en lugar de agregar.

**Severidad**: рҹ”ҙ вҖ” afecta a TODO mensaje enviado.

### #6 вҖ” NotificationsView infinite loading

**ReproducciГіn**: navegar a `/app/notifications` o `/company/notifications`.

**Causa**: `NotificationsView.jsx:60` usa `state.currentUser?.id` (AppContext) en vez de `user?.id` (AuthContext). El AppContext nunca dispatcha `SET_USER` en el flujo actual, asГӯ que `userId` es `null` y el `load()` retorna early sin setear `loading=false`.

**Fix**: cambiar a `const { user } = useAuth()` siguiendo el patrГіn de Chat.jsx, MatchesView.jsx, etc. CompanyNotificationsView tiene el mismo bug.

**Severidad**: рҹ”ҙ вҖ” vista crГӯtica completamente inutilizable.

### #7 вҖ” AuthContext lock AbortError bloquea rutas privadas tras navegaciГіn

**ReproducciГіn**: hacer login вҶ’ navegar a otra vista вҶ’ refrescar вҶ’ ir a `/company/stats` o `/company/create-offer` o cualquier ruta privada.

**SГӯntoma**: spinner infinito. Console muestra:
```
[error] [AuthContext] Session init error: AbortError: Lock broken by another request with the 'steal' option.
```

**Causa**: `supabase.auth.getUser()` usa internamente un `LockManager` (Web Locks API). Con HMR de Vite + StrictMode + mГәltiples requests concurrentes (auth init + onAuthStateChange + cualquier query con sesiГіn), el lock falla con `AbortError`. El catch en `AuthContext.jsx:24-26` solo loguea el error y NO setea `user`, ni hace `setLoading(false)`. Resultado: `PrivateRoute` espera para siempre.

**Fix**:
- En `AuthContext.initSession`, fallback a `supabase.auth.getSession()` si `getUser()` falla con `AbortError`.
- O configurar el cliente en `lib/supabase.js`: `createClient(url, key, { auth: { storage: localStorage, persistSession: true, autoRefreshToken: true } })` y considerar deshabilitar `lock` browser-level.
- Asegurar que el `finally { setLoading(false) }` se ejecute aГәn en error.

**Severidad**: рҹ”ҙ вҖ” bloquea CompanyStats, CreateOffer y cualquier ruta privada tras la primera navegaciГіn. La Гәnica razГіn por la que CompanyDashboard cargГі fue porque fue la primera ruta tras login (sin reload de por medio).

**ReproducciГіn**: pegar texto > 2000 chars en input de chat y enviar.

**SГӯntoma**: el backend correctamente retorna error y `useMessages.js` revierte el optimistic. PERO el usuario no recibe feedback вҖ” solo console.error.

**Severidad**: рҹҹ  вҖ” la validaciГіn funciona, pero la UX es muda.
