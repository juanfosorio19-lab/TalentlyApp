## Cambios aplicados en esta sesión

### Issue #1 — useSwipeProfiles.js
Cambio defensivo: convertir `swipedIds` (que podía ser undefined) en un `Set` con fallback a array vacío. Mejora también performance del `.filter()` (Set lookup O(1) vs Array.includes O(n)).

### Issue #2 — Chat.jsx
Manejo explícito de errores en `db.matches.get()` y `db.profiles.getById()`. Si fallan, se setea `otherProfile = { full_name: 'Usuario' }` para que la vista no quede en blanco.

### Issue #3 — Charts tooltips
- Nuevas variables en `variables.css`: `--chart-tooltip-bg`, `--chart-tooltip-text`, `--chart-point-border` (con valores invertidos en dark mode).
- `BarChart.jsx` y `LineChart.jsx` ahora leen estas variables via `getPropertyValue()` siguiendo el patrón ya establecido para `--primary`, `--text-secondary`, etc.

### Issue #4 — CompanyFiltersView
Alineadas todas las classNames a `fv__*` para que coincidan con `FiltersView.css` (que ya importaba). Antes la vista renderizaba sin estilos. Cambios:
- `filters-view` → `fv`
- `filters-view__header` → `fv__header`
- `filters-view__back` → `fv__close`
- `filters-view__title` → `fv__title`
- `filters-view__clear` → `fv__reset`
- `filters-view__scroll` → `fv__scroll`
- `filters-section__label` → `fv__section-label`
- `filters-chips` → `fv__chips`
- `filters-chip` → `fv__chip` (+ `--active`)
- `filters-select` → `fv__select`
- `filters-view__footer` → `fv__footer` (+ agregado `fv__btn-clear` para parity)
- `filters-view__apply` → `fv__btn-apply`

## Cambios adicionales en la 2da pasada (issues #6, #7, #8, #11)

### Issue #6 — useMessages.js
- `db.statistics.increment('messages_sent')` ahora dentro de try/catch (no crítico, no debe romper UX de envío).
- Si `db.matches.sendMessage` falla, ahora hay `return` temprano para no contabilizar stats de un mensaje que no se envió.

### Issue #7 — AuthContext.jsx
- Flag `isMounted` agregado dentro del `useEffect`. Tanto `initSession` como el callback de `onAuthStateChange` verifican `isMounted` antes de cada `setState` (3 puntos críticos). Cleanup setea `isMounted = false`.

### Issue #8 — Step7_PosicionesSeniority.jsx
- Eliminados 60+ líneas de mappings `POSITION_ICONS` y `SENIORITY_YEARS`.
- Aplicada **migración 012** (`company_positions_seniority_metadata.sql`):
  - Columna `icon` agregada a `company_positions` y populada con Material Symbols.
  - Columna `years_range` agregada a `seniority_levels` y populada con strings display.
- El JSX ahora lee `p.icon` y `s.years_range` directamente. Si la BD agrega nuevas posiciones, basta con setear `icon` en el INSERT — no hace falta tocar JSX.

### Issue #11 — Columnas explícitas en queries de perfiles ajenos
- Dos constantes nuevas en `src/lib/supabase.js`: `PROFILE_PUBLIC_COLS` (~40 columnas, sin `email`, `birthday`, `birth_date`, `latitude`, `longitude`, `notification_prefs`) y `COMPANY_PUBLIC_COLS` (sin `tax_id`, coords, notification_prefs).
- Nuevo método `db.profiles.getPublicById(id)` para consultar perfiles de OTROS usuarios.
- Queries actualizadas:
  - `db.profiles.getDiscovery` y `getCandidatesForExplore` → `PROFILE_PUBLIC_COLS`
  - `db.matches.getWithProfiles` (perfiles del otro lado) → `PROFILE_PUBLIC_COLS`
  - `db.offers.getById` (join companies) → `COMPANY_PUBLIC_COLS`
- Vistas actualizadas:
  - `Chat.jsx` → `getPublicById(otherId)`
  - `CompanyPublicProfileView.jsx` (candidato viendo empresa) → `getPublicById`
  - `CandidatePublicProfileView.jsx` (empresa viendo candidato) → `getPublicById`
- `db.profiles.getById` se mantiene con `*` porque sus callers son el propio user (AuthContext, ProfileView edit, onboarding, etc.) — necesitan ver/editar `birthday`, `birth_date`, etc.
